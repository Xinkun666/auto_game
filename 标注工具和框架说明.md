1. GameAuto Studio 使用说明文档
	1.1. 工具简介
GameAuto Studio 是一款专为游戏自动化测试/脚本开发设计的可视化标注工具。它支持通过 hdc实时抓取设备画面，或通过本地图片进行标注。标注结果可一键导出为标准化的 Python 数据结构，极大简化了模板匹配和像素坐标的管理。

直接运行项目文件里的aw/autogame/
tools/Label.py 即可运行标注工具

	1.2. 核心概念
在开始标注前，请理解以下三级结构：
● 阶段 (Stage)：对应游戏的长流程（如：主界面、副本中、设置页）。
● 场景 (Scene)：阶段内的具体画面（如：副本中的结算画面、战斗画面）。
● 项目 (Item)：场景内的具体标注对象：
		■ 区域 (Area)：用于模板匹配的特征图，支持设置搜索范围 (Search Scope)。
		■ 控点 (Control)：用于点击或交互的具体坐标点。
		■ 特殊区域 (Special Area)：用于特殊逻辑处理，支持归一化坐标导出。

通过这种方式，一个游戏自动化的逻辑就划分为很多个阶段，每个阶段里又会有很多场景，每个场景里又会有对应的区域、控点和特殊区域等。
● 区域的目的是在游戏自动化时，判断你框选区域内的图片特征是否出现。
● 控点的目的是要对该控点执行对应的操作，包括点击、长按、按下、拖拽、双击，有多个控点可以实现双指等。
● 特殊区域的目的是对于有些区域，我们并不是想要获取他是否出现，而是这个区域图片更深层的信息，比如游戏小地图我们想要获取的是人物位置信息，游戏方位我们想要获取的是人物方位信息，前景的障碍物是想要获取障碍物信息。因此特殊区域的处理函数也是开发者自定义的。

	1.3. 功能指南
1.3.1 项目管理
1. 新建项目：通过菜单栏 文件 -> 新建项目 开始。
2. 导入项目：支持导入已有的工程目录（需包含 info.py），方便二次修改。
3. 项目树：右侧树状图显示 项目 -> 阶段 -> 场景 -> 标注项 的层级关系。
1.3.2 图像获取
 选择一个场景后，在操作面板点击：
1. 抓图：通过 hdc 命令自动获取连接设备的实时画面（路径：/data/local/tmp/）。
2. 导入图片：从本地磁盘选择已有的游戏截图。
1.3.3 标注操作
1. 添加区域 (Area)：点击按钮后在画布上拖拽。完成后会生成特征小图。
		■ 操作：点击按钮后，在画面中框选具有代表性的图标、按钮或特征。
		■ 产物：系统会自动裁剪该矩形区域并保存为 templates 下的小图，用于后续的模板匹配。
		■ 搜索范围 (Search Scope)：
			● 核心功能：选中已有的 Area 项，点击“设置搜索范围”。
			● 建议：设置搜索范围旨在对局部图像进行图像特征判定，从而节省时间并提高识别准确率。如果一个图标出现的位置是动态的，则可以设置较大的搜索范围来保证定位的准确度。
2. 添加控点(Control)：
		■ 操作：点击按钮后，在目标位置画一个小框（通常框选按钮的中心区域）。
		■ 用途：主要记录一个交互坐标。导出的数据中包含该框的中心点或矩形范围，自动化脚本可以直接调用该坐标执行 click 或 tap 操作。
		■ 表现：在画布上以红色实线框表示。
3. 添加特殊区域(Special)
	■ 操作：点击按钮并在目标位置框选。
	■ 高级设置：
		◆ 归一化范围：支持通过“设置区域范围”手动输入 x1,y1,x2,y2（0-1 之间）。
		◆ 代码联动：导出项目时，系统会在 SpecialSceneHandler.py 中自动生成以该区域命名的函数。开发者可以将 OCR 文字识别、颜色统计、图像处理等复杂逻辑写在对应的函数里。
	■ 表现：在画布上以橙色实线框表示。
1.3.4 画布交互：
1. 缩放：Ctrl + 鼠标滚轮 或使用顶部的“放大/缩小”按钮。
2. 适配：点击“适配”使图片完整显示在当前窗口。
3. 十字准星：实时显示当前鼠标在原图上的绝对像素坐标。
1.3.5 导出与集成
点击 文件 -> 导出项目：
● 资源生成：在目标目录生成 scenes（完整截图）和 templates（特征小图）文件夹。
● 代码生成：生成 info.py，包含所有坐标、路径及归一化数据的字典。
● 特殊处理句柄：自动在 resource/SpecialSceneHander.py 中生成特殊区域的空函数，方便开发者直接编写业务代码。
● 导出时，代码会直接导出到正确位置，不需要人为修改！！！

2. 自动化脚本开发指南
在完成标注工具的资源导出后，你需要编写具体的业务逻辑脚本（即 on_stage 函数），让引擎知道在特定的画面下该执行什么操作。
2.1 环境准备与目录规范
为了保证自动化引擎能正确识别你的脚本与资源，必须严格遵守以下目录结构：
1. 资源存放：将标注工具导出的整个项目文件夹（包含 info.py、scenes/、templates/）存放在框架的 customs_examples/ 目录下。
2. 脚本存放：在 customs_game_examples/ 目录下创建一个同名文件夹。在该文件夹内新建一个 .py 文件撰写脚本。
3. 命名限制：目录名请务必使用英文（如 genshin_login），避免使用中文导致路径编码解析异常。
目录示例：
ProjectRoot/
├── customs_examples/
│   └── my_game_project/          # 导出的资源目录
│       ├── info.py
│       └── templates/
└── customs_game_examples/
    └── my_game_project/          # 脚本逻辑目录
        └── login_logic.py        # 你的自动化脚本
2.2 核心 API 说明书
在编写 on_stage(w: 'FrameWorker') 时，你可以通过参数 w 调用以下核心方法：
2.2.1 状态与流程控制
● w.current_stage：获取当前生效的阶段名（对应 info.py 中 STAGE_DICT 值为 True 的键）。
● w.change_stage('阶段名')：切换当前逻辑阶段。
● w.refresh_frame()：强制刷新当前帧信息（重新进行图像识别），常用于点击后的画面确认。
● w.stop()：停止整个自动化流程。
2.2.2 图像识别与交互
● w.get_info(area_name)：核心判定函数。返回该标注区域（Area）是否在当前画面中出现（布尔值）。
该函数是整个脚本的“眼睛”，用于判断标注的特征是否存在于当前画面。
● 参数说明：
	○ area_name (str): 你在标注工具中定义的 Area（区域） 名称。
● 返回值：
	○ bool: 若匹配度超过预设阈值（默认 0.7）则返回 (x,y)(该区域中心点在屏幕上的相对位置)，否则返回 False。
● 开发建议：
	○ 常用于 if 条件判断。如果识别不灵敏，请检查标注工具中 Area 的范围是否包含太多背景。
● w.click(btn_name, x_bias=0, y_bias=0)：点击对应的“控点”中心。
执行最常用的单次模拟点击操作。
● 参数说明：
	○ btn_name (str): 标注好的 Control（控点）的名称。
	○ x_bias / y_bias (int): 像素偏移量。正值代表向右/向下偏移。
● 动作原理：
	○ 引擎会自动计算该标注框的 中心点坐标，叠加偏移量后发送 uinput -c (Click) 指令。
● 代码示例：
w.click('返回按钮', x_bias=-10) # 点击 返回按钮 中心左侧10像素的位置
● w.tap_single(btn, wait=100, dura=500, x_bias=0, y_bias=0)：单指模拟操作，支持像素偏移。
这个函数模拟的是一个从起点到终点的位移触控过程。
● 执行逻辑：
		i. 起点：标注项 btn 的中心点 (X, Y)。
		ii. 按下并滑动：从起点开始按下，滑动到偏移位置 (X+x\_bias, Y+y\_bias)(向上滑时y_bias是负数，向下滑时，y_bias是正数，向右滑时,x_bias是正数，向左滑时,x_bias是负数)
		iii. 滑动耗时：这个位移过程持续 dura 毫秒。
		iv. 抬起等待：到达目标点后，手指保持按下状态 wait 毫秒后才抬起。
● 典型应用：
		i. 短距离滑动：如设置 y_bias=-100 模拟小幅度上滑。
		ii. 精准长按：设置 x_bias=0, y_bias=1（微小偏移）且 wait=2000，模拟在某处按住 2 秒。
● w.tap_double(btn1, btn2, wait=100, dura=500, x1_bias=0, y1_bias=0, x2_bias=0, y2_bias=0)：双指同时模拟操作，常用于缩放或多点交互。
● 执行逻辑：
	○ 两个手指分别从 btn1 和 btn2 的中心点同时按下。
	○ 各自根据设定的 bias 偏移量进行滑动，滑动过程持续 dura 毫秒。
	○ 到达各自目标点后，同步保持 wait 毫秒后抬起。
● 典型应用：
	○ 地图缩放：设置两个手指朝相反方向位移（一个 bias 为正，一个为负）。
	○ 双指并行滑动：在某些需要双指同步操作的游戏机制中使用。
● w.click_down(btn, x_bias=0, y_bias=0, dura=0)：按下不松开。
发送“按下”指令但不立即抬起。
● 参数说明：
	○ dura: 如果设为 0，则只发送 down 事件（手指停留在屏幕上）。如果设为大于 0 的值，则会在指定时间后执行 up（抬起）。
● 用途：
	○ 摇杆控制：配合偏移量实现角色持续行走。
	○ 特殊蓄力：模拟游戏中需要蓄力触发的技能按键。
2.2.3 日志与分析
insert_logs(任务名, 持续时间, *关键词)：记录测试数据。
● 例如：insert_logs('登录测试', 30, '帧率', '插帧')
● 此操作会记录时间区间，并配合 time.sleep(30) 使用，后续分析模块会据此提取日志中的 FPS 等数据。
2.3 脚本开发示例
以下是一个标准的脚本结构，展示了如何利用识别到的画面元素进行逻辑跳转。
import time
from aw.autogame.tools.Utils import insert_logs
if TYPE_CHECKING:
    from aw.autogame.tools.GameFrameWorker import FrameWorker

def on_stage(w: 'FrameWorker'):
    """
    每帧都会回调此函数。
    w: FrameWorker 实例，提供操作 API
    """
    
    # 1. 判断当前所处的阶段（由 info.py 定义）
    if w.current_stage == '登录界面':
        
        # 2. 识别画面中的 Area（区域）
        if w.get_info('确认按钮'):
            print("检测到确认按钮，准备进入游戏...")
            
            # 3. 执行点击操作
            w.click('确认按钮')
            
            # 4. 操作后通常需要等待并刷新帧信息
            time.sleep(2)
            w.refresh_frame() 

    elif w.current_stage == '大厅':
        
        if w.get_info('活动弹窗'):
            w.click('关闭按钮')
            time.sleep(2)
            w.refresh_frame()
            
        # 示例：记录 10 秒的性能数据，并标记该段逻辑涉及“帧率”分析
        if w.get_info('进入战斗'):
            insert_logs('战斗测试', 10, "帧率")
            w.click('进入战斗')
            w.change_stage('战斗中') # 切换逻辑阶段
            
        if w.get_info('退出'):
            w.stop() # 结束自动化
