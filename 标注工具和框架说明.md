# 标注工具和自动化框架上手指南

这份文档面向第一次接触 `auto_game` 的开发者。目标不是把每个源码细节都讲完，而是让你知道：

1. 这套框架解决什么问题。
2. 用标注工具应该标什么、导出什么。
3. 业务脚本应该写在哪里。
4. 怎么把资源、脚本和启动入口串起来。
5. 出问题时优先检查哪里。

如果你只想最快跑通一个最小闭环，请先看第 2 节。

## 1. 先理解整体架构

这套工程把游戏自动化拆成三层：

| 层级 | 目录/文件 | 职责 |
| --- | --- | --- |
| 标注资源 | `aw/autogame/customs_examples/<project_case>/` | 存放标注工具导出的 `info.py`、截图、模板图、特殊区域处理函数 |
| 业务逻辑 | `aw/autogame/customs_game_examples/<project_case>/<target_case>.py` | 编写每一帧看到什么、点击什么、切换到哪个阶段 |
| 启动入口 | `main.py`、`launcher.py`、`testcases/` | 设置环境变量、启动游戏、启动自动化、收集日志和结果 |

运行时的大致流程是：

```text
标注工具导出 info.py
        |
        v
GameAutomator 启动视频流和 FrameWorker
        |
        v
FrameWorker 按当前阶段识别 Area / Special Area
        |
        v
每一帧调用 customs_game_examples 里的 on_stage(w)
        |
        v
业务脚本通过 w.click / w.tap_single / w.change_stage 等接口执行动作
```

你可以简单记成：

- `customs_examples` 是“眼睛和地图”。
- `customs_game_examples` 是“决策逻辑”。
- `testcases`、`main.py`、`launcher.py` 是“启动方式”。

## 2. 5 分钟跑通最小闭环

最短路径如下：

1. 确认设备已连接，并且命令行能执行：

```bash
hdc list targets
```

2. 启动标注工具：

```bash
python aw/autogame/tools/Label.py
```

3. 在标注工具中新建项目，例如 `MyGameDemo`。
4. 新建一个阶段，例如 `登录阶段`。
5. 新建一个场景，抓图或导入一张游戏截图。
6. 在截图上标一个 `Area`，例如 `确认按钮特征`。
7. 在截图上标一个 `Control`，例如 `确认按钮`。
8. 点击 `文件 -> 导出项目`，导出到默认的 `aw/autogame/customs_examples/`。
9. 新建业务脚本：

```text
aw/autogame/customs_game_examples/MyGameDemo/demo.py
```

10. 写一个最小 `on_stage(w)`：

```python
import time


def on_stage(w):
    if w.current_stage == "登录阶段":
        if w.get_info("确认按钮特征"):
            w.click("确认按钮")
            time.sleep(1)
            w.refresh_frame()
```

11. 在启动入口中设置：

```python
project_case = "MyGameDemo"
target_case = "demo"
os.environ["TARGET_PROJECT_CASE"] = project_case
os.environ["TARGET_GAME_CASE"] = target_case
```

12. 运行 `main.py` 或使用 `launcher.py` 选择项目启动。

跑通这个闭环后，再逐步增加阶段、场景、特殊区域和复杂逻辑。

## 3. 两个名字必须对上

框架靠两个环境变量定位资源和业务脚本：

| 名称 | 含义 | 对应路径 |
| --- | --- | --- |
| `TARGET_PROJECT_CASE` / `project_case` | 标注工具导出的资源工程名 | `aw/autogame/customs_examples/<project_case>/info.py` |
| `TARGET_GAME_CASE` / `target_case` | 业务逻辑脚本名 | `aw/autogame/customs_game_examples/<project_case>/<target_case>.py` |

例如：

```python
project_case = "Auto_PUBG_ALL"
target_case = "auto_pubg"
```

框架会加载：

```text
aw/autogame/customs_examples/Auto_PUBG_ALL/info.py
aw/autogame/customs_game_examples/Auto_PUBG_ALL/auto_pubg.py
```

新手最常见的启动失败，基本都是这两个名字、目录名或文件名没对上。

## 4. 推荐目录结构

不要随意移动框架目录。推荐保持下面的结构：

```text
auto_game/
├── main.py
├── launcher.py
├── testcases/
├── aw/
│   └── autogame/
│       ├── customs_examples/
│       │   └── <project_case>/
│       │       ├── info.py
│       │       ├── scenes/
│       │       ├── templates/
│       │       └── resource/
│       │           └── SpecialSceneHandler.py
│       ├── customs_game_examples/
│       │   └── <project_case>/
│       │       └── <target_case>.py
│       ├── tools/
│       │   ├── Label.py
│       │   ├── GameAutomator.py
│       │   ├── GameFrameWorker.py
│       │   └── GameSceneHandler.py
│       └── temp/
```

各目录职责：

- `customs_examples/<project_case>`：标注工具导出的资源，不建议手工改 `info.py`。
- `customs_examples/<project_case>/resource`：特殊区域代码、模型、算法工具等资源。
- `customs_game_examples/<project_case>`：你写自动化业务逻辑的地方。
- `testcases`：接入 SP、启动游戏、抓日志、性能分析时使用。
- `aw/autogame/temp`：运行时日志、帧图、分析结果和归档文件。

## 5. 标注工具怎么用

启动：

```bash
python aw/autogame/tools/Label.py
```

标注工具里有三层数据：

| 概念 | 说明 | 示例 |
| --- | --- | --- |
| `Stage` 阶段 | 自动化流程中的大状态 | 登录阶段、跑图阶段、开车阶段 |
| `Scene` 场景 | 某个阶段下的一张具体画面 | 主界面、弹窗、地图界面 |
| `Item` 标注项 | 场景中的识别或操作对象 | Area、Control、Special Area |

### 5.1 Area：用于判断特征是否出现

`Area` 会被导出成模板图，运行时通过模板匹配判断当前画面里是否出现。

适合标：

- 按钮图标
- 固定文字提示
- 状态图标
- 不容易变化的 UI 特征

建议：

- 框选范围尽量小而稳定，少带背景。
- 特征可能移动时，给 Area 设置 `Search Scope`。
- 同一个阶段内，Area 名称尽量唯一，避免脚本里 `w.get_info("名称")` 取到不符合预期的结果。

运行时：

```python
if w.get_info("确认按钮特征"):
    ...
```

匹配成功时通常返回当前屏幕上的归一化坐标 `(x, y)`；失败时返回 `False` 或空值。

### 5.2 Control：用于执行点击、滑动、长按

`Control` 表示一个可操作位置。框架会取标注框中心点作为操作点。

适合标：

- 按钮中心
- 摇杆中心
- 视角滑动区域的起点
- 地图缩放的两个手指起点

常用脚本：

```python
w.click("确认按钮")
w.tap_single("摇杆", y_bias=-300, dura=300, wait=1000)
```

### 5.3 Special Area：用于自定义图像处理

`Special Area` 不是简单判断“有没有出现”，而是把区域图片交给你写的函数处理。

适合做：

- 小地图定位
- 人物朝向识别
- 障碍物检测
- OCR 文字识别
- YOLO / 分类模型推理

导出后，标注工具会在下面文件里生成同名函数：

```text
aw/autogame/customs_examples/<project_case>/resource/SpecialSceneHandler.py
```

例如你标了一个特殊区域叫 `location`，就需要在 `SpecialSceneHandler.py` 中实现：

```python
def location(img):
    # img 是当前帧裁剪出的特殊区域图像
    return ...
```

脚本里读取结果：

```python
pos = w.get_info("location")
if pos:
    ...
```

## 6. 标注工具常用操作

项目操作：

- `文件 -> 新建项目`：创建新标注项目。
- `文件 -> 导入项目`：导入已有工程目录，目录里需要有 `info.py`。
- `文件 -> 导出项目`：导出到 `customs_examples`。

图像操作：

- `抓图`：通过 `hdc` 从当前设备抓取画面。
- `导入图片`：用本地截图进行标注。
- 重新抓图或重新导入不同尺寸图片时，已有标注会按图片宽高比例重新换算位置。

快捷操作：

- `Ctrl + A`：添加 Area。
- `Ctrl + C`：添加 Control。
- `Ctrl + S`：添加 Special Area。
- `Ctrl + 鼠标滚轮`：缩放画布。
- 鼠标右键：打开当前对象菜单。
- 拖动标注框：调整位置。

导出策略：

- 导出时会生成 `info.py`、`scenes/`、`templates/`、`resource/`。
- 选择“替换原工程”时，会扫描导出瞬间旧工程的 `resource/`，避免标注期间新增资源被误删。
- 替换时只同步变更文件，未变化文件会跳过。
- 如果导出失败，会回滚本次同步过程中改动过的文件。

## 7. 导出结果是什么

导出后的目录大致如下：

```text
<project_case>/
├── info.py
├── scenes/
├── templates/
└── resource/
    └── SpecialSceneHandler.py
```

`info.py` 里最重要的是：

- `PROJECT_NAME`：工程名。
- `STAGE_DICT`：哪些阶段存在，以及当前默认激活哪个阶段。
- `STAGE_INFO`：每个阶段、场景、Area、Control、Special Area 的配置。

业务脚本一般不直接改 `info.py`，而是通过 `w.get_info(...)`、`w.click(...)` 等接口读取和使用它。

## 8. 业务脚本怎么写

业务脚本位置：

```text
aw/autogame/customs_game_examples/<project_case>/<target_case>.py
```

必须提供：

```python
def on_stage(w):
    ...
```

`FrameWorker` 每一帧都会调用一次 `on_stage(w)`。推荐写法是：

1. 先判断当前阶段 `w.current_stage`。
2. 再用 `w.get_info(...)` 判断当前画面。
3. 执行动作。
4. 动作后适当 `time.sleep(...)`。
5. 调用 `w.refresh_frame()` 更新识别结果。
6. 必要时用 `w.change_stage(...)` 切换阶段。

示例：

```python
import time
from typing import TYPE_CHECKING

if TYPE_CHECKING:
    from aw.autogame.tools.GameFrameWorker import FrameWorker


def on_stage(w: "FrameWorker"):
    if w.current_stage == "登录阶段":
        if w.get_info("确认按钮特征"):
            w.click("确认按钮")
            time.sleep(1)
            w.refresh_frame()

        if w.get_info("进入大厅标志"):
            w.change_stage("大厅阶段")

    elif w.current_stage == "大厅阶段":
        if w.get_info("开始按钮特征"):
            w.click("开始按钮")
            time.sleep(2)
            w.refresh_frame()

        if w.get_info("退出按钮特征"):
            w.stop()
```

## 9. 常用 API 速查

### 9.1 流程控制

| API | 作用 |
| --- | --- |
| `w.current_stage` | 当前激活阶段名 |
| `w.change_stage("阶段名")` | 切换当前阶段 |
| `w.refresh_frame()` | 重新取一帧并重新识别 |
| `w.stop()` | 停止自动化 |

### 9.2 识别接口

```python
w.get_info("标注名称")
```

说明：

- 对 `Area`：匹配成功时返回归一化位置 `(x, y)`，失败时返回 `False`。
- 对 `Special Area`：返回你在 `SpecialSceneHandler.py` 对应函数里的返回值。
- 常用写法是把它当条件判断：

```python
if w.get_info("确认按钮特征"):
    w.click("确认按钮")
```

### 9.3 点击和滑动

```python
w.click("按钮名", x_bias=0, y_bias=0)
```

点击 `Control` 中心点，可叠加像素偏移。

```python
w.tap_single("摇杆", x_bias=0, y_bias=-300, dura=300, wait=1000)
```

单指操作语义：

1. 从 `Control` 中心按下。
2. 在 `dura` 毫秒内滑动到偏移后的目标点。
3. 到达目标点后继续按住 `wait` 毫秒。
4. 默认松手；如果 `release=False`，则保持按下。

常见用法：

```python
# 角色向上推摇杆，滑动 300ms，到位后继续按住 1000ms 再松手
w.tap_single("摇杆", y_bias=-300, dura=300, wait=1000)

# 视角向右滑
w.tap_single("视角", x_bias=500, dura=400, wait=100)
```

```python
w.tap_double(
    "手指1",
    "手指2",
    x1_bias=-200,
    y1_bias=0,
    x2_bias=200,
    y2_bias=0,
    dura=300,
    wait=100,
)
```

双指操作语义和 `tap_single` 一致：两根手指同时滑到目标点，再等待，再按配置松手。

```python
w.click_down("按钮名", dura=0)
```

按下操作：

- `dura=0`：只按下，不自动松开。
- `dura>0`：按住指定时间后松开。

### 9.4 sendevent 专用连续手势

当使用 `sendevent` 后端时，还可以直接控制手指生命周期：

```python
w.move_press(0, "摇杆")
w.move_to(0, "摇杆", y_bias=-300, duration_ms=300)
time.sleep(1)
w.move_up(0)
```

这适合需要跨多个逻辑步骤保持按下的场景。

默认触控后端由 `TARGET_TOUCH_BACKEND` 控制，当前框架默认使用 `sendevent`。

## 10. SpecialSceneHandler 怎么写

`SpecialSceneHandler.py` 是特殊区域的业务处理文件。每个特殊区域对应一个同名函数。

示例：

```python
def direction(img):
    return dire_tool.get_direction(img)


def location(img):
    return loc_tool.get_location(img)
```

注意：

- 函数名要和 Special Area 名称对应。
- `img` 是裁剪后的区域图像，不是整张屏幕。
- 函数返回什么，`w.get_info("特殊区域名")` 就拿到什么。
- 模型初始化可以放在文件顶部，但要注意启动耗时和依赖是否完整。

## 11. 怎么启动

### 11.1 用 main.py 直接联调

适合你已经手动打开游戏，只想快速调业务脚本。

在 `main.py` 中确认：

```python
DEFAULT_PROJECT_CASE = "Auto_PUBG_ALL"
DEFAULT_TARGET_CASE = "auto_pubg"
```

然后运行：

```bash
python main.py
```

默认模式是 `direct`。如果要走 testcase：

```bash
AUTOGAME_MAIN_MODE=testcase python main.py
```

### 11.2 用 launcher.py 图形化启动

适合手动选择项目、选择 testcase、配置运行次数、观察实时预览。

```bash
python launcher.py
```

### 11.3 用 testcases 接完整测试流程

适合需要：

- 启动游戏或性能工具。
- 接 SP / xdevice。
- 抓设备日志。
- 跑完后分析性能结果。
- 做清理和归档。

testcase 中必须设置：

```python
project_case = "Auto_PUBG_ALL"
target_case = "auto_pubg"

os.environ["TARGET_PROJECT_CASE"] = project_case
os.environ["TARGET_GAME_CASE"] = target_case
```

同时注意：

- testcase 文件名和类名要一致。
- 对应 `.json` 配置要存在。
- `target_case` 要能在 `customs_game_examples/<project_case>/` 下找到同名 `.py` 文件。

## 13. 常见问题排查

### 13.1 提示 `TARGET_PROJECT_CASE 未设置`

说明启动前没有设置资源工程名。检查：

```python
os.environ["TARGET_PROJECT_CASE"] = project_case
```

### 13.2 提示找不到业务逻辑

检查三件事：

```text
aw/autogame/customs_game_examples/<project_case>/<target_case>.py
```

- `<project_case>` 是否和环境变量一致。
- `<target_case>.py` 文件名是否和环境变量一致。
- 文件里是否有 `def on_stage(w):`。

### 13.3 `w.get_info(...)` 一直失败

优先检查：

- Area 是否框得太大，带了太多背景。
- Search Scope 是否过小，目标稍微移动就超出范围。
- 当前阶段是否正确。
- Area 名称是否写错。
- 截图分辨率和当前设备画面差异是否过大。

### 13.4 点击位置不对

优先检查：

- Control 是否标在按钮中心。
- 当前设备旋转方向是否稳定。
- `x_bias` / `y_bias` 是否方向写反。
- 是否正在使用 `sendevent`，以及设备触摸坐标映射是否正确。

坐标偏移约定：

- `x_bias > 0`：向右。
- `x_bias < 0`：向左。
- `y_bias > 0`：向下。
- `y_bias < 0`：向上。

### 13.5 滑动后没有按住

`tap_single` 的语义是：

```text
按下起点 -> 用 dura 滑到目标点 -> 在目标点 wait -> 松手
```

如果你希望滑到目标点后不松手，传：

```python
w.tap_single("摇杆", y_bias=-300, dura=300, wait=1000, release=False)
```

或者使用 `move_press / move_to / move_up` 拆开控制。

### 13.6 Special Area 没有结果

检查：

- `SpecialSceneHandler.py` 里是否有对应函数。
- 函数名是否和特殊区域名称一致。
- 函数内部依赖是否安装，例如模型库、OCR 库等。
- 函数是否真的 `return` 了结果。

### 13.7 抓图或触控失败

先检查：

```bash
hdc list targets
```

再检查：

- 设备是否在线。
- `hdc` 是否在 `PATH`。
- 设备权限是否允许截屏和输入事件。
- `sendevent` 依赖的工具是否已准备好。

## 14. 推荐开发顺序

新手不要一开始就写完整自动化。推荐顺序：

1. 只做一个阶段、一个场景、一个 Area、一个 Control。
2. 写一个只会点击按钮的 `on_stage(w)`。
3. 用 `main.py` 跑通。
4. 加入第二个阶段，练习 `w.change_stage(...)`。
5. 加入 `Search Scope`，优化识别速度和稳定性。
6. 加入 `Special Area`，实现位置、方向、OCR 或模型推理。
7. 最后再接 `testcases`、日志抓取、性能分析和归档。

## 15. 提交代码时建议提交什么

通常只需要提交和你的项目相关的目录：

```text
aw/autogame/customs_examples/<project_case>/
aw/autogame/customs_game_examples/<project_case>/
testcases/<你的用例目录>/
```

如果你改了框架能力，例如 `Label.py`、`GameFrameWorker.py`、`GameSceneHandler.py`，再提交对应框架文件。

提交前建议检查：

- 标注资源是否已经导出。
- `SpecialSceneHandler.py` 中依赖的模型或资源路径是否存在。
- `target_case` 是否能被正确导入。
- 本地临时结果目录 `aw/autogame/temp/` 不要误提交。

## 16. 一句话总结

这套框架的正确使用方式是：

```text
用 Label.py 标注并导出资源
       ->
在 customs_game_examples 写 on_stage(w)
       ->
用 main.py / launcher.py / testcases 启动
       ->
根据 temp 下的帧图、日志和结果不断调试
```

先跑通最小闭环，再逐步加复杂能力，效率最高。
